# `ecosystem-integration` Code Review

CR-1 | CRITICAL | CLI server commands wrong in builtin rule
Rule listed `server start/stop` but actual CLI has `server mcp start/stop`
and `server service start/stop/status/warmup`. LLMs following this rule
would get "No such command" errors.
**Status:** FIXED — updated to full server subcommand tree.

CR-2 | CRITICAL | search_codebase MCP signature incomplete
Rule omitted `function_name` and `class_name` parameters from
`search_codebase` tool. These are real filter params in `mcp_server.py:547-548`.
**Status:** FIXED — added both parameters to rule.

CR-3 | CRITICAL | Default port wrong
Rule said 8765, `config.py:89` says 8766.
**Status:** FIXED — corrected to 8766.

CR-4 | HIGH | Missing server service subcommands
`server service status` and `server service warmup` omitted from CLI
commands section.
**Status:** FIXED — added full server mcp and server service command sets.

CR-5 | HIGH | Missing env vars from EnvVar enum
Rule lists 4 env vars but `EnvVar` enum defines 8 RAG-specific vars.
Internal vars (`QDRANT_DIR`, `INDEX_META`, `CODE_INDEX_META`, `STATUS_DIR`)
omitted. Acceptable — these are internal implementation details not useful
for LLM guidance. Rule documents the user-facing subset.
**Status:** ACCEPTED — by design.

CR-6 | MEDIUM | Frontmatter name divergence between source and synced copy
Source has `name: vaultspec-rag`, synced copy has `name: vaultspec-rag.builtin`
with `trigger: always_on`. This is core sync's transform behavior — it appends
`.builtin` suffix and adds trigger field. Not a bug.
**Status:** ACCEPTED — core sync behavior.

CR-7 | MEDIUM | CLI command table style inconsistent with vaultspec-cli.builtin.md
Rule used flat list instead of grouped sections with headers.
**Status:** FIXED — grouped into "Search and indexing", "Server management",
"Development" sections matching core rule style.

C1 | CRITICAL | .gitignore managed block too broad
`.vault/` and `.vaultspec/` in the `vaultspec-managed` block ignore all vault
documents and framework content. However, this block is generated by
`vaultspec-core sync` and marked "do not edit". Already-tracked files remain
in the index. New vault docs require `git add -f`.
**Status:** KNOWN LIMITATION — core's design. Removed duplicate manual entries
above the managed block to reduce confusion. Recommend filing core issue.

C2 | MEDIUM | Duplicate .gitignore entries
8 entries appeared both in manual section and managed block.
**Status:** FIXED — removed manual duplicates, managed block is authoritative.

H1 | HIGH | CLAUDE.md self-ignored by managed block
`CLAUDE.md` is committed but also in the managed block's ignore list.
Existing clones unaffected (file is in index). Fresh `git add CLAUDE.md`
would be ignored without `-f`.
**Status:** KNOWN LIMITATION — same core sync behavior as C1. CLAUDE.md is
managed by core; core expects to control tracking.

R1 | MEDIUM | ADR missing plan back-link
ADR `related:` field did not include `[[2026-04-06-ecosystem-integration-plan]]`.
**Status:** FIXED — added back-link.

R2 | LOW | Plan has non-standard `revised:` frontmatter field
`revised: 2026-04-11` not in vault template schema. Revision date already
documented in heading and body text.
**Status:** ACCEPTED — provides useful metadata, no harm.

L1 | LOW | pymarkdown.json change minimal and correct
Only `vaultspec` appended to `allowed_elements`. Required for CLAUDE.md.
**Status:** PASS.
