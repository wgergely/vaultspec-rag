## Audit Findings

### 1. Custom Snapping Grid System

The custom snapping grid system is primarily implemented in `legacy/system/window/src/layout/grid.rs` and is visualized by the `mainwindow-snappoint` crate.

#### Core Components

- **`SnapGrid` struct**: This is the central component for defining and managing the grid. It is initialized with a `ScreenRect` (the screen area it covers) and a `GridConfig`.
  - **Creation**: `SnapGrid::new(area: ScreenRect, config: GridConfig)`
  - **Density Calculation**: The `cell_width` and `cell_height` (which determine grid density) are calculated in `SnapGrid::new` based on the `area`'s dimensions and the `cols` and `rows` specified in `GridConfig`:

        ```rust
        let cell_width = area.width / config.cols.max(1) as f32;
        let cell_height = area.height / config.rows.max(1) as f32;
        ```

        A higher number of `cols`/`rows` results in smaller cells and a denser grid.
- **`GridConfig` struct**: (Imported from `super::spacing::GridConfig`) This struct configures the grid's structure and snapping behavior:
  - `cols`: Number of horizontal divisions.
  - `rows`: Number of vertical divisions.
  - `snap_to_cell_centers`: A boolean indicating whether snapping should also consider cell centers.
  - `snap_threshold`: An `f32` value (in pixels) that defines the maximum distance a window can be from a snap point (or cell center) to trigger snapping. The `docs/design/window_management.md` states this is "approx 40% of spacing".
- **`GridPoint` struct**: Represents an intersection point on the grid. It contains:
  - `col: u32`, `row: u32`: The grid coordinates (0 to `cols`/`rows`).
  - `x: f32`, `y: f32`: The absolute screen coordinates in pixels.
  - **Creation**: `GridPoint::from_grid(col, row, origin, cell_width, cell_height)` translates grid indices to pixel coordinates:

        ```rust
        x: origin.x + col as f32 * cell_width,
        y: origin.y + row as f32 * cell_height,
        ```

#### Snapping Logic

- **`SnapGrid::nearest_point_xy(x, y)`**: Finds the `GridPoint` closest to a given `(x, y)` coordinate and returns the point and the distance to it. This uses rounding of relative coordinates to `col` and `row` indices.
- **`SnapGrid::nearest_cell_center_xy(x, y)`**: Similar to `nearest_point_xy`, but targets the center of the nearest grid cell.
- **`SnapGrid::snap_position(x, y)`**: The core snapping decision logic.
  - It compares distances to the `nearest_point` and `nearest_cell_center` (if `snap_to_cell_centers` is true).
  - If the closest candidate's distance (`best_dist`) is less than or equal to `self.config.snap_threshold`, it returns `Some((snapped_x, snapped_y))`; otherwise, `None`.
- **`SnapGrid::snap_window_xy(x, y, width, height)`**: Adapts `snap_position` for entire windows.
  - It first attempts to snap the window's top-left corner using `snap_position`.
  - Crucially, it then clamps the final `x` and `y` coordinates to ensure the entire window `(x, y, width, height)` remains fully within the `SnapGrid`'s defined `area`.

#### Visualization of Snap Points (`mainwindow-snappoint`)

- **`SnapPointWindow`**: Defined in `legacy/mainwindow/snappoint/src/window.rs`, this struct manages the display of grid points. It holds a `SnapPointConfig` and a `Vec<GridPoint>`.
- **`SnapPointWindow::show(&mut self, points: Vec<GridPoint>)`**: This method takes a `Vec<GridPoint>` (generated by the `SnapGrid` logic) and stores them for rendering.
- **`SnapPointConfig`**: (in `legacy/mainwindow/snappoint/src/config.rs`) Configures the *visuals* of the snap points (e.g., `mode` (Circle/Square), `base_size_px`, `color`, `fade_duration_secs`). It does not define how the grid points are calculated, only how they appear.
- **`render_points` function**: (in `legacy/mainwindow/snappoint/src/renderer.rs`) Responsible for drawing the `GridPoint`s onto a pixel buffer based on the `SnapPointConfig`.

### 2. Window Management Logic (Resizing, Dragging, Positioning)

The `mainwindow-frame` crate (`legacy/mainwindow/frame/src/lib.rs`) provides the fundamental logic for handling frameless window interactions like resizing and dragging, leveraging the `egui` framework.

#### Core Components

- **`MainWindow` struct**: Manages the state of window interactions (`drag_active`, `resize_active`, `centered`, etc.).
- **`RESIZE_BORDER: f32 = 6.0`**: Defines the pixel width of "hot zones" at window edges for resize detection.
- **`DRAG_HANDLE_HEIGHT: f32 = 20.0`**: Defines the height of an invisible region at the top of the window that acts as a draggable area.

#### Resizing Logic

- **`detect_edges(pos: Pos2, window_rect: Rect, border: f32) -> EdgeProximity`**: A pure function that determines which window edges a given mouse `pos` is near, within the specified `border`.
- **`EdgeProximity` struct**: Stores boolean flags for `left`, `right`, `top`, `bottom` proximity.
- **`EdgeProximity::resize_direction()`**: Translates `EdgeProximity` into an `egui::viewport::ResizeDirection` (e.g., `NorthWest`, `East`) and an appropriate `egui::CursorIcon` for visual feedback.
- **`MainWindow::handle_resize(&mut self, ctx: &Context)`**:
  - Continuously checks mouse position relative to window edges using `detect_edges`.
  - If an edge zone is active and a mouse button is pressed, it sets `self.resize_active = true`, stores the `resize_direction`, and sends an `egui::ViewportCommand::BeginResize(direction)` to the `egui::Context`. This command initiates a native resize operation through the OS.
  - `resize_active` is reset when the mouse button is released.

#### Dragging Logic

- **`MainWindow::render_drag_handle(&mut self, ui: &mut Ui, ctx: &Context)`**:
  - Creates an `egui::Sense::click_and_drag()` interaction area covering the `DRAG_HANDLE_HEIGHT` at the top of the window.
  - When this area is dragged (`response.dragged()` is true), it sends an `egui::ViewportCommand::StartDrag` to the `egui::Context`. This command instructs the OS to start a native window drag operation.
  - `drag_active` state is updated based on `response.drag_started()` and `response.drag_stopped()`.

#### Positioning Logic (Centering)

- **`calculate_centered_position(monitor_size: Vec2, window_size: Vec2) -> Pos2`**: A utility function to compute the top-left coordinates for centering `window_size` within `monitor_size`. It handles cases where the window is larger than the monitor by clamping coordinates to `0.0`.
- **`MainWindow::center_on_show(&mut self, ctx: &Context)`**:
  - Called once on initial window show.
  - Retrieves current monitor and window sizes via `egui::Context`.
  - Uses `calculate_centered_position` to determine the target position.
  - Sends an `egui::ViewportCommand::OuterPosition(pos)` to reposition the window.

### 3. Transparency or Custom Frame Implementation

- **Custom Frame**: The `mainwindow-frame` crate explicitly implements custom frame behavior by providing the logic for hit-testing resize borders and a drag handle, relying on `egui`'s `ViewportCommand`s to interface with the native windowing system for the actual resize and drag operations. This means the OS window manager is effectively delegated control for the visual manipulation, but *how* the frameless window is rendered (e.g., shadows, rounded corners, custom title bar) is left to the `egui` application itself.
- **Transparency**: No explicit code or configuration related to window transparency (e.g., setting alpha channels for the window itself, or blur effects) was found within the `legacy/mainwindow/frame` crate or the `mainwindow-snappoint` crate. It is likely that transparency, if implemented, would be handled at a higher level during the initial window creation or through specific platform-dependent APIs, and not within these core logic components.
